# commit

변경 내용을 분석해 Conventional Commits 규격의 커밋 메시지(한국어)를 작성하고,
안전 규칙을 지키며 실제 커밋까지 실행한다.

## 워크플로우

### 1. 변경 분석
- `git status --porcelain`로 전체 상태를 본다.
- 스테이징된 변경이 있으면 `git diff --staged`를, 없으면 `git diff`를 읽는다.
- diff를 실제로 읽고 "무엇을 / 왜" 바꿨는지 파악한다.
- 성격이 다른 변경이 섞여 있으면, 알려주고 나눠서 커밋할지 물어본다.
  (원칙: 커밋 하나에는 논리적으로 하나의 변경만.)

### 2. 스테이징 (필요할 때만)
- 스테이징된 게 없거나 다르게 묶어야 하면 명시적으로 스테이징한다.
  - 특정 파일: `git add 경로/파일`
  - 패턴: `git add src/components/*`
  - 부분 스테이징: `git add -p`
- **비밀정보는 절대 커밋하지 않는다**: `.env`, `credentials.json`,
  개인 키, 토큰 등이 보이면 멈추고 사용자에게 알린다.

### 3. 메시지 작성
아래 "메시지 규칙"에 따라 작성한다. 작성한 메시지를 먼저 사용자에게 보여준다.

### 4. 커밋 실행
- 한 줄:
  ```
  git commit -m "<type>(<scope>): <요약>"
  ```
- 본문/푸터가 있으면 HEREDOC:
  ```
  git commit -m "$(cat <<'EOF'
  <type>(<scope>): <요약>

  <본문>

  <푸터>
  EOF
  )"
  ```
- 커밋 후 `git log -1 --stat`으로 결과를 간단히 확인시켜 준다.
- 요청받지 않으면 `git push`는 하지 않는다.

## 메시지 규칙

### 제목 줄
- 형식: `<type>(<scope>): <요약>` — `<scope>`는 선택.
- Type: `feat`, `fix`, `refactor`, `perf`, `docs`, `test`, `chore`,
  `build`, `ci`, `style`, `revert` (type/scope 키워드는 영어 그대로).
- 요약은 **한국어**로 간결하게. 무엇을 했는지 동작 중심으로 쓴다.
  (예: "리프레시 토큰 회전 추가", "빈 커서 처리", "쓸모없는 로그 제거")
- 가능하면 50자 이내, 최대 72자.
- 끝에 마침표 없음.
- 콜론 뒤 대소문자/표기는 프로젝트 관례를 따른다.

### 본문 (필요할 때만)
- 제목만으로 자명하면 본문은 생략한다.
- 다음 경우엔 본문을 넣는다: 이유가 자명하지 않을 때, 파괴적 변경,
  마이그레이션 안내, 연결된 이슈.
- 한 줄 72자에서 줄바꿈.
- 불릿은 `*`가 아니라 `-`를 쓴다.
- 이슈/PR 참조는 맨 끝에: `Closes #42`, `Refs #17`.

### 반드시 본문을 넣어야 하는 경우 (Auto-Clarity)
파괴적 변경, 보안 수정, 데이터 마이그레이션, 이전 커밋 되돌리기(revert)는
**절대 제목만으로 압축하지 않는다.** 나중에 디버깅할 사람에게 맥락이 필요하다.

### 파괴적 변경 (Breaking Change)
- type/scope 뒤에 `!`: `feat(api)!: /v1/orders를 /v1/checkout으로 변경`
- 그리고 본문/푸터에 `BREAKING CHANGE:` 로 상세 설명.

### 절대 넣지 않는 것
- "이 커밋은 ~한다", "나는/우리는", "지금/현재" 같은 표현 — diff가 무엇을
  바꿨는지 이미 말해준다.
- "~의 요청으로" 같은 문구 — 필요하면 `Co-authored-by` 푸터를 쓴다.
- "Generated with Claude Code" 등 AI 표기 — 단, 프로젝트 규칙이
  `Assisted-by` 같은 AI 표기 푸터를 요구하면 그때만 푸터로 추가.
- 이모지 (프로젝트 관례가 요구하는 경우 제외).
- scope에 이미 나온 모듈/파일명을 요약에서 반복하기.

## Git 안전 규칙
- git config를 **절대** 수정하지 않는다.
- `--force`, hard reset 등 파괴적 명령은 명시적 요청 없이 실행하지 않는다.
- `--no-verify`로 훅을 건너뛰지 않는다 (사용자가 요청한 경우 제외).
- main/master로 force push 하지 않는다.
- 훅 때문에 커밋이 실패하면, 문제를 고친 뒤 **새 커밋**을 만든다 (amend 금지).

## 모드
- 기본은 간결(terse) 스타일: 자명한 변경은 제목 한 줄.
- "verbose mode" 또는 "자세히"라고 하면, 본문에 맥락을 더 풍부하게 담는다.

## 예시

프로필 조회 엔드포인트 추가 (이유를 본문에 설명):
```
feat(api): GET /users/:id/profile 추가

모바일 클라이언트가 콜드런치 화면에서 LTE 대역폭을 아끼기 위해
전체 user 페이로드 없이 프로필 데이터만 받아야 한다.

Closes #128
```

파괴적 API 변경:
```
feat(api)!: /v1/orders를 /v1/checkout으로 변경

BREAKING CHANGE: /v1/orders를 쓰던 클라이언트는 2026-06-01 전에
/v1/checkout으로 이전해야 한다. 이후 기존 경로는 410을 반환한다.
```

자명한 변경 (본문 생략):
```
fix(auth): 빈 리프레시 토큰에서 나던 500 에러 처리
```
